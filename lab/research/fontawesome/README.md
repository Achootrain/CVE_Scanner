# Font Awesome WP-plugin CVE research lab

Targeted vulnerability fingerprinting lab. **Not** about the core Font Awesome
icon library by FortAwesome -- that library has 0 NVD CVEs and 0 GitHub
Security Advisories. Every CVE that mentions "font awesome" is against a
third-party WordPress plugin that bundles or wraps Font Awesome (Better Font
Awesome, Block for Font Awesome, AGP Font Awesome Collection, etc.).

The lab catalogs those CVEs, resolves each plugin's WordPress.org slug, builds
a detection rule for every slug (path + regex + vulnerable-range), and validates
the rules against the 71 FA-positive Vietnamese sites already captured in
`data/scan_results.jsonl`.

## Files

```
lab/research/
  mine_paths.py            -- generic library-agnostic version-mining script
                              (driven by per-library/plugin config.yaml)
  mine_all_plugins.py      -- driver: walks slugs.json active plugins,
                              auto-generates per-plugin configs via wp.org API
                              and SVN tag listing, runs mine_paths.py per plugin

lab/research/fontawesome/
  README.md                -- this file
  config.yaml              -- library config: repo, versions, path globs, regex bank
  version_rules.json       -- 28 library-source rules (FA CSS/JS bundles)
  cves.json                -- NVD keyword search 'font awesome', structured per CVE
  slugs.json               -- wordpress.org slug per CVE plugin
  3_build_rules.py         -- transform: cves.json + slugs.json -> rules.json
  4_validate.py            -- live-fetch rules against FA-positive sites
  rules.json               -- 18 vulnerable WP plugins, 24 CVEs (CVE-driven)
  validation.md            -- human-readable scan report
  out/                     -- gitignored; raw per-run artifacts

lab/research/plugins/
  _summary.json            -- per-plugin mining outcome (rule count, primary, cross)
  <slug>/
    config.yaml            -- generated by mine_all_plugins.py
    version_rules.json     -- per-plugin detection rules (PHP header + readme + ...)
    out/                   -- gitignored; downloaded plugin zip + intermediate data
```

## Two pipelines, two rule sets

### A. CVE-driven WP-plugin detection (`rules.json`)

```
NVD (keywordSearch=font+awesome)  --> cves.json   (24 entries)
wordpress.org/plugins/<slug>/     --> slugs.json  (18 plugins)
3_build_rules.py merges them      --> rules.json  (18 rules)
4_validate.py probes 69 targets   --> validation.md
```

Inputs (`cves.json`, `slugs.json`) were collected by a one-shot WebFetch +
subagent fan-out at lab bootstrap. To refresh, re-fetch and overwrite in
place, then re-run `3_build_rules.py`. Hand-edit also fine for new CVEs.

### B. Library version-detection mining (`version_rules.json`)

```
FortAwesome/Font-Awesome github source  --> out/path_matches.json
   (download primary + scan for version)
raw.githubusercontent fetch per path    --> out/cross_validate.json
   (confirm each path leaks version at older cross-versions too)
regex bank match per stable path        --> version_rules.json  (28 rules)
```

Run via the generic miner:

```bash
python lab/research/mine_paths.py --config lab/research/fontawesome/config.yaml

# Override primary / cross versions without editing config:
python lab/research/mine_paths.py --config lab/research/fontawesome/config.yaml \
    --primary 8.0.0 --cross 7.2.0,6.7.2
```

The same `mine_paths.py` works for any GitHub-hosted JS/CSS library --
add a sibling `lab/research/<libname>/config.yaml` declaring the repo,
include path globs, and regex bank, then point the script at it.

### C. WordPress plugin mining (`plugins/<slug>/version_rules.json`)

The same `mine_paths.py` engine, but with `archive_url` / `raw_url` pointing
at wordpress.org distribution endpoints rather than github.com:

```yaml
# plugins/better-font-awesome/config.yaml (auto-generated)
slug: better-font-awesome
archive_url: "https://downloads.wordpress.org/plugin/{slug}.{tag}.zip"
raw_url:     "https://plugins.svn.wordpress.org/{slug}/tags/{tag}/{path}"
primary: "2.0.4"
cross: ["2.0.3", "2.0.2", "2.0.1"]
include_paths:
  - "readme.txt"
  - "*.php"
  - "**/*.php"
  - "block.json"
regex_bank:
  - label: "wp plugin readme: Stable tag: X.Y.Z"
    pattern: '(?im)^Stable\s*tag\s*:\s*([0-9][\w.\-]*)'
  - label: "php header: Version: X.Y.Z"
    pattern: '(?im)^\s*\*?\s*Version\s*:\s*([0-9][\w.\-]*)'
```

The driver (`mine_all_plugins.py`) generates these configs automatically for
every active plugin in `slugs.json`. SVN tags are listed via the SVN HTTP
directory page; the wp.org plugin API supplies the current version for
trunk-only plugins (plugins whose authors commit directly without tagging).

Run end-to-end:

```bash
python lab/research/mine_all_plugins.py
# -> plugins/<slug>/version_rules.json for each of 10 active plugins
# -> plugins/_summary.json aggregate
```

Yields 16 detection rules across 10 plugins (vs 10 rules in the hand-built
CVE map -- 1.6x more probe paths). The empirical scan also surfaces
non-obvious main-PHP filenames (e.g. `/index.php`, `/fa-icons.php`,
`/eds_font_awesome.php`) that a hand-written rule set would miss.

## Rule shape

```json
{
  "slug": "block-for-font-awesome",
  "product": "Block for Font Awesome",
  "wp_status": "active",
  "paths": ["/wp-content/plugins/block-for-font-awesome/readme.txt"],
  "version_regex": "(?im)^Stable\\s*tag:\\s*([0-9][\\w.\\-]*)",
  "cves": [
    {"cve": "CVE-2023-49751", "cvss": 8.8, "vuln_type": "CSRF",
     "affected": "<= 1.4.0", "range": {"op": "le", "version": "1.4.0"}, ...},
    {"cve": "CVE-2024-35705", "cvss": 5.4, "vuln_type": "XSS",
     "affected": "< 1.4.5",  "range": {"op": "lt", "version": "1.4.5"}, ...}
  ]
}
```

The detection mechanic relies on a universal WordPress convention: every
plugin ships a `readme.txt` whose header has `Stable tag: <ver>`. If the host
returns that file for a given slug, the plugin is installed; we then compare
the extracted version against each CVE's `range` to mark vulnerable / patched.

## Why this works

Of the 71 FA-positive sites in `scan_results.jsonl`, **zero** carry a version
today because the prior scan only probed the single nuclei path
`/wp-content/plugins/font-awesome/readme.txt` (the official FortAwesome plugin
slug). The new catalog adds 17 alternative slugs to probe -- those are the
*vulnerable* installs the prior scan missed entirely.

## Validation modes

```bash
# Full sweep (71 targets x 18 rules = 1278 probes; ~15 min sequential)
python lab/research/fontawesome/4_validate.py --concurrency 1

# Smoke test on a few targets
python lab/research/fontawesome/4_validate.py --max-targets 3 --concurrency 6

# Custom target list
python lab/research/fontawesome/4_validate.py --targets example.com foo.vn
```

The validator uses per-host serial dispatch when `concurrency >= 2` (no two
in-flight probes against the same host). At `concurrency 1` it is fully
sequential. Timeout is 4s per probe; soft-404 200 responses (HTML SPA shells)
are rejected by a header heuristic before regex extraction.

## Known limitations

- **Universal soft-404s.** Some hosts serve 200/HTML for every unknown path.
  The "`===` or `Stable tag` in first 200 bytes" filter catches most of these,
  but a host that injects "Stable tag" into a custom 404 template would slip
  through. False-positive count is reported in `validation.md`.
- **CDN-served FA.** Many sites load Font Awesome from a CDN
  (`use.fontawesome.com`, `cdnjs.cloudflare.com`) and have no plugin installed
  at all. Those sites correctly report 0 plugin hits.
- **Closed slugs still on disk.** 10 of the 18 plugins were removed from
  wordpress.org (security closure or author request), but installed sites
  continue to serve `readme.txt` until upgraded -- those are the highest-value
  detections, since no upstream patch is coming.
- **Same CVE, different slug.** Two plugins share the name "Font Awesome
  Integration" but have different vendors (FortAwesome vs. unrelated author);
  CVE-2023-5233 is against the unrelated one. Slug-based matching disambiguates
  correctly because the slugs differ.
